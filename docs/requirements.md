# DB Mock 需求规格

状态：已确认，可进入实现  
更新时间：2026-07-22

## 1. 产品目标

DB Mock 面向个人及公司内部团队。用户在 Web 页面上选择数据库类型、版本和目标
主机，平台通过 SSH 在目标主机上生成并执行 Docker Compose 项目，完成数据库实例
的完整生命周期管理。

平台只部署拥有可用容器镜像的数据库，不登记或代理既有数据库连接，也不提供 SQL
控制台、表浏览、数据编辑或数据库内部账号管理。

## 2. 首版范围

### 2.1 支持

- Linux：Ubuntu、Debian、Rocky Linux、AlmaLinux，`amd64` 与 `arm64`。
- macOS：必须预装且运行 Docker Desktop，并开启 Remote Login/SSH。
- IPv4；不要求 IPv6。
- 最多 10 台主机、30 名用户、50 个数据库实例。
- 单机实例；一个实例只属于一台主机。复杂数据库允许在同一主机内运行多个容器。
- SSH 直连，不支持跳板机。
- SSH 密码和私钥认证，首次连接确认并保存主机指纹。
- Linux 可选择由平台安装/升级 Docker；此时 SSH 用户必须具备免密 `sudo`。
- HTTP/HTTPS 代理、私有仓库凭据和私有仓库自签名 CA。
- 平台默认 HTTP；配置证书后由同一个 Go 服务直接提供 HTTPS。

### 2.2 不支持

- Windows。
- 跨主机数据库集群、故障数据迁移和自动主从切换。
- 跨主机备份、对象存储和备份下载。
- 平台元数据备份。
- SQL 控制台、表结构/数据浏览与数据库内部账号管理。
- LDAP、AD、OIDC、SSO 和角色权限体系。
- 面向第三方自动化的稳定公开 API 与长期 API Token。
- 自动修改操作系统防火墙或云厂商安全组。

## 3. 用户与安全

- 第一次启动进入初始化页面创建首个账号，关闭公开注册。
- 所有账号拥有相同权限，均可查看和操作全部项目、主机、实例、镜像、模板、连接
  凭据、用户和审计日志。
- 任意登录用户可新增、禁用和重置其他用户；不能禁用或删除最后一个可登录用户。
- 不设置密码长度、复杂度和失败锁定策略；密码仍必须非空并使用 Argon2id 哈希保存。
- 登录会话默认有效 30 天，Cookie 使用 HttpOnly、SameSite 和 Secure（HTTPS 时）。
- SSH、镜像仓库和数据库凭据使用平台主密钥加密保存。
- 自定义 Compose 与脚本被视为可信代码。平台展示危险能力风险报告但允许使用
  `privileged`、宿主机挂载和 host 网络等配置，所有执行均写审计日志。
- 审计日志默认永久保留，可检索、导出 CSV 和按时间范围手动清理；清理动作本身保留。

## 4. 项目与资源组织

- 实例可属于一个项目，并选择开发、测试、预发布或生产环境。
- 支持任意标签；项目、环境和标签只用于筛选、统计及 Webhook，不影响权限。
- 项目中仍有主机或实例时禁止删除。
- 主机同时保存 SSH 管理地址与数据库连接地址。
- 实例内部标识和模板版本不可变；显示名称、项目、环境与标签可以修改。

## 5. 主机管理

接入主机时填写名称、SSH 地址/端口/用户、认证方式、数据库连接地址、数据根目录、
端口池、代理及是否允许管理 Docker。平台检测：

- SSH 与主机指纹。
- 操作系统、CPU 架构、CPU、内存、磁盘。
- Docker Engine 与 Compose v2 版本及 Docker 可用资源。
- 数据根目录可写性和端口池。

连接测试成功后由服务端签发 10 分钟有效的验证凭证，凭证绑定 SSH 地址、端口、用户、
认证凭据、数据根目录、端口池和实际主机指纹。新增主机以及修改这些运行参数时必须提交
匹配的验证凭证，不能只由前端声明“已经测试”。已接入主机发生指纹变化时拒绝连接。
探测会实际创建并删除权限为当前 SSH 用户的临时文件以确认数据根目录可写，并使用
`ss`、`lsof` 或 `netstat` 查找端口池中的首个空闲端口。

默认端口池为 `20000-40000`。默认数据根目录由操作系统决定并允许修改。平台只管理
该目录下由自身创建的实例，不接管已有容器。

无法访问软件源的离线 Linux 主机必须预装 Docker。Docker 安装/升级必须由用户手动
触发，不做无人值守自动升级。

主机有实例或平台管理的数据时禁止删除。SSH 不可达时只标记离线，不删除记录。
维护模式暂停该主机的主动检测和告警。
主机仍有实例或备份时禁止修改数据根目录；缩小端口池时必须继续覆盖全部现有实例端口。

## 6. 资源调度

- 创建实例时根据架构兼容性、模板最低要求和主机实时剩余资源自动推荐主机，允许
  用户手动选择。
- 默认保留主机 10% CPU、20% 内存和 20% 磁盘。
- 支持按数据库模板定义小型、中型、大型资源预设，也支持自定义 CPU、内存和预留磁盘。
- 资源不足或端口冲突时拒绝创建。
- 磁盘容量用于调度与告警，不跨操作系统强制磁盘配额；80% 警告，90% 严重告警。

## 7. 数据库实例生命周期

创建向导：

1. 选择模板和固定版本。
2. 自动推荐或手动选择主机。
3. 填写显示名称、项目、环境、标签和数据库初始化参数。
4. 选择资源预设或自定义资源、端口及高级环境变量/启动参数。
5. 设置异常自动重启。
6. 检查变更计划并创建，进入任务中心查看实时进度。

实例支持创建、启动、停止、重启、查看连接信息、日志和监控、修改可变配置、升级和
删除。可变运行配置包括 CPU、内存、预留磁盘和额外环境变量：变更以持久化任务执行，
入队时原子预留目标容量，运行中实例短暂重建容器，已停止实例只校验 Compose 而不启动；
执行失败时恢复原 Compose、资源配置和运行/停止状态。数据目录、端口、初始化参数和
数据库凭据不随运行配置变更。数据库密码默认自动生成并允许自定义，加密保存并可再次查看。连接信息包含地址、
端口、用户名、密码、URI 及适用时的 JDBC 地址。

异常自动重启支持主机默认值和实例级开关，默认开启。手动停止的实例不得自动拉起。
异常连续重启失败 3 次后停止重试并告警。

数据库补丁或同主版本升级允许创建内部临时数据快照，失败时回滚，成功后立即删除。
主版本升级首版不支持，除非模板包含经过验证的专属迁移流程。
升级时可为目标版本重新选择直接拉取、匹配的私有仓库凭据或包含目标镜像且与实例主机
架构兼容的离线镜像；成功后实例记住新的镜像来源，失败回滚时保留原来源。

实例支持手动冷备份、恢复和删除。备份保存在实例主机 `<data-root>/backups/`
下，不会在控制节点保留第二份。运行中实例会短暂停机以创建一致归档，完成后恢复
原期望状态。平台记录归档大小和 SHA-256；恢复前重新校验，并创建临时回滚快照。
恢复后容器无法通过健康检查时自动恢复操作前数据。为避免跨数据库版本损坏，备份只能恢复到
原实例且实例必须仍使用备份时的模板版本。平台使用主机上已存在的实例镜像，以禁网、根文件
系统只读的临时容器访问由数据库 UID 持有的数据，不在备份流程中拉取额外镜像。

每个实例可配置每日或每周自动冷备份，计划包含执行时刻、IANA 时区和保留份数。
到期计划仅在实例处于运行中或已停止状态时原子创建备份与任务；实例正执行其他操作时
保持到期并在恢复稳定后补跑，不抢占或中断生命周期任务。每次成功后按当前策略清理超出
保留份数的计划备份，人工备份永不自动删除。修改或关闭策略后，已排队任务完成时重新读取
当前策略，避免使用过期保留值。自动创建与清理均进入任务中心、审计和 Webhook 链路。

删除必须输入实例名称二次确认。删除容器、Compose 网络、数据卷和平台数据目录，保留
Docker 镜像供其他实例复用。外部路径永不由平台递归删除。

## 8. 模板目录

### 8.1 标准模板

MySQL、MariaDB、PostgreSQL、Redis、Valkey、MongoDB、ClickHouse、Elasticsearch、
OpenSearch、InfluxDB、Neo4j、Cassandra、Microsoft SQL Server、Oracle Database Free、
openGauss。

标准模板提供创建、启停、异常重启、删除、日志、监控、连接信息和同主版本受控升级。

### 8.2 实验模板

TiDB、OceanBase CE、StarRocks、Apache Doris。实验模板在一台主机内运行所需的多个
容器，页面明确标记为开发/测试用途。

### 8.3 明确排除

Greenplum、DB2 z/OS、DB2 IBM i、GaussDB、PolarDB、AnalyticDB、Hologres、
MaxCompute 及其他没有可自由部署容器镜像的云/商业数据源不作为内置模板。

### 8.4 自定义模板包

上传 `.zip`，包含 `dbmock-template.yaml`、`docker-compose.yml`、图标、配置和可选脚本。
模板包定义参数表单、资源要求、连接信息、健康检查、宿主机调优、升级协议和支持架构。
没有升级脚本的自定义模板只能使用镜像重建式同主版本升级。需要备份与回滚的持久化数据必须
绑定到 `{{ .DataPath }}` 下，`spec.image` 必须提供 `/bin/sh` 与 `tar`。

模板采用不可变版本；编辑创建新版本，既有实例保持原版本。被实例引用的模板或镜像
禁止删除。

## 9. 镜像与仓库

- 配置多个公有或私有 OCI/Docker Registry，凭据加密保存。
- 支持代理、自签名 CA，并通过 SSH 安装 CA 后重启 Docker（仅用户明确触发）。
- 支持上传 `docker save` 生成的 `.tar` 或 `.tar.gz`。
- 文件分片上传、断点续传、SHA-256 校验、按镜像摘要去重；默认单文件上限 50 GB，
  可在设置中调整。
- 离线镜像集中保存在控制平台，通过可续传的 SSH/SFTP 分发到主机并执行 `docker load`。
- 实例创建和升级均可选择离线镜像；排队或执行中的任务引用期间禁止清理对应归档或仓库。
- 提供未使用镜像扫描和手动清理；不自动删除主机镜像。

## 10. 监控、日志与告警

- 每 30 秒采集主机和实例的 CPU、内存、磁盘、容器状态及健康检查，历史保存 7 天。
- 容器日志轮转：单文件 100 MB，保留 5 个；网页实时查看并下载。
- 默认告警：主机连续 3 次不可达、容器异常退出、容器健康检查失败、连续重启失败、磁盘 80%/90%、
  升级失败、SSH 凭据失效。
- 用户可调整阈值、关闭单项告警和将主机置为维护模式。
- 通用 Webhook 支持多个地址、事件筛选、HMAC 签名、投递记录和手动重投；业务事件失败
  最多重试 5 次，测试请求只发送 1 次。禁用 Webhook 时取消尚未发送的投递，并拒绝测试
  和重投操作。
- 告警确认与解决记录处理账号和时间；自动恢复的告警记录系统处理人，重复处理返回状态冲突。

## 11. 任务中心与审计

创建、删除、运行配置变更、升级、备份/恢复、镜像上传/分发和 Docker 安装/升级均为持久化异步任务。任务展示
状态、阶段、进度、实时日志、耗时和错误；仅在安全阶段允许取消，失败后允许重试。
应用重启后恢复排队任务，并把无法继续的运行中任务标记为中断。

审计记录用户、来源 IP、动作、资源、脱敏前后差异、结果、任务和时间。

## 12. 页面

初始化、登录、总览、项目、主机、数据库目录、实例列表与详情、镜像与仓库、自定义
模板、任务中心、告警与 Webhook、用户、审计、系统设置。

页面支持简体中文与英文。采用浅色响应式管理后台；重点适配桌面和平板，手机端保证
基本查看但不优化复杂创建流程。支持最近两个大版本的 Chrome、Edge、Firefox 和 Safari。

## 13. 部署与开源交付

- Apache-2.0 许可证，不收集遥测数据。
- 单个 Go 应用镜像内嵌 React 静态资源；不需要单独前端容器或 Nginx。
- Docker Compose 包含应用和 PostgreSQL 元数据服务。
- 应用镜像提供 `linux/amd64` 与 `linux/arm64`。
- 控制平台最低配置：2 CPU、4 GB 内存、20 GB 可用磁盘。
- 离线 `.tar.gz` 包含应用镜像、PostgreSQL 镜像、Compose、环境变量示例、安装/升级
  脚本和中英文部署文档；数据库镜像由用户另行上传或从私有仓库拉取。
- GitHub Actions 执行单元测试、后端测试、SSH/Docker 模拟集成测试、前端端到端测试、
  多架构构建和安全检查。

## 14. 验收标准

1. 新环境可通过在线或离线 Compose 启动，并在首次访问完成初始化。
2. Linux x86 主机可通过密码或私钥接入、检测资源，并按选择安装 Docker。
3. 可使用一个标准模板及一个自定义模板完整创建、查看连接信息、启停、监控、升级和删除。
4. 离线镜像可以断点上传、校验、分发并用于实例创建。
5. 资源不足、端口冲突、SSH 失效、容器异常和磁盘阈值均能被识别并按配置发送 Webhook。
6. 中英文切换、30 天会话、用户管理、任务恢复、审计检索/导出/清理均可用。
7. `cd backend && go test ./...`、前端类型检查与测试、生产构建和
   `deploy/compose.yaml` 健康检查通过。
8. 可为运行中或已停止的实例创建主机本地冷备份；恢复前校验归档完整性，恢复失败时
   自动回滚，删除备份后主机文件与平台元数据均被清理。
9. 可在容量与模板下限允许时调整实例 CPU、内存、预留磁盘和额外环境变量；运行中实例
   成功后保持运行，已停止实例保持停止，失败时自动恢复原配置且任务历史不泄露环境变量值。
10. 可按实例配置每日或每周自动冷备份、时区和保留份数；实例忙碌时安全延后，计划备份
    成功后只清理超出保留数的计划备份，人工备份不受影响，失败任务和最近状态可追踪。
